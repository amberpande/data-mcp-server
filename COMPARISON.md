## 1. Architecture

| Aspect | SnowMCP | Snowflake Cortex Analyst |
|--------|---------|--------------------------|
| Protocol | Model Context Protocol (MCP) — open standard by Anthropic | Proprietary REST API + Snowflake Cortex runtime |
| Server location | Your infrastructure (local, Docker, Kubernetes) | Snowflake-managed (runs inside Snowflake) |
| Client | Any MCP-compatible client: Claude Code, Ollama, custom agents | Snowflake UI, Streamlit in Snowflake, REST API consumers |
| Transport | stdio (local) or HTTP + SSE (remote) | HTTPS REST API |
| Stateful? | In-memory dataset cache per session | Stateless per request |
| Open source | Yes — full source code | No — closed, managed service |

## 3. Semantic Layer

| Aspect | SnowMCP | Snowflake Cortex Analyst |
|--------|---------|--------------------------|
| Format | YAML entity files (`schemas/*.yaml`) | YAML semantic model stored in Snowflake Stage |
| Schema concepts | Entities, metrics, attributes, join paths, filters | Tables, dimensions, measures, time dimensions, verified queries |
| Metric definitions | `aggregation` + `source_field` + optional `filter` | `expr` (SQL expression) — more flexible |
| Verified / sample queries | Not built-in (can add via prompt) | Built-in `verified_queries` block — pre-validated SQL pinned |
| Version control | Git — full history, PRs, branch strategies | File in Snowflake Stage — no native git integration |
| Schema validation | Pydantic model validation on startup | Snowflake validates on upload |
| Multi-hop joins | Manual YAML relationship chains | Automatic join resolution via semantic graph |
| Calculated metrics | Via `filter` field on aggregations | Full SQL `expr` — any SQL expression as a measure |
| Schema discoverability | `get_semantic_context` MCP tool | `/api/v2/cortex/analyst/message` parses intent |

**Cortex Analyst edge:** `verified_queries` let you pin exact SQL for known questions — guarantees those specific questions are always answered identically. SnowMCP doesn't have this yet.

**SnowMCP edge:** Schema lives in git, goes through code review, can be tested in CI, and works across Snowflake + local data simultaneously.

## 4. SQL Generation Quality

| Aspect | SnowMCP | Snowflake Cortex Analyst |
|--------|---------|--------------------------|
| Grounding mechanism | `lookup_metric` tool + schema in system prompt | Semantic model constrains SQL generator |
| Column hallucination risk | Low — schema injected at session start | Very low — constrained to declared columns |
| Metric consistency | High — canonical formulas in YAML | Very high — exact SQL expressions in YAML |
| Join correctness | YAML join paths + system prompt rules | Automatic join planning from semantic graph |
| Snowflake-specific syntax | Explicit rules: `QUALIFY`, `IFF`, `APPROX_COUNT_DISTINCT`, `NULLIF`, `DATE_TRUNC` | Generated by Snowflake's own model — native Snowflake SQL |
| Pre-execution validation | `validate_sql` tool runs `EXPLAIN USING TABULAR` | Built-in — Cortex validates before returning |
| Complex query handling | Depends on LLM capability — Claude 4 handles well | Fan-out / chasm trap detection built into Cortex |
| Confidence scoring | Not built-in | Cortex returns a `confidence` field per response |

**Cortex Analyst edge:** Native Snowflake SQL generation, automatic fan-out/chasm trap detection, and confidence scoring are baked in. No prompt engineering needed.

**SnowMCP edge:** Full transparency — you can see, edit, and test the SQL rules. With Claude 4, generation quality is on par or better for complex analytical questions.

## 8. Cost

### 8e. Key cost levers

| Lever | SnowMCP | Cortex Analyst |
|-------|---------|----------------|
| Switch to cheaper model instantly | ✅ Haiku 4 at 10× cheaper than Sonnet | ❌ No model choice |
| Use Ollama for zero LLM cost | ✅ | ❌ |
| Cache repeated queries in-memory | ✅ DATASETS dict (CSV), extend for SF | ❌ |
| Auto-scale infra down at night | ✅ K8s HPA / scheduled scaling | ❌ Serverless (billed per query anyway) |
| Pin to lower Snowflake tier | ✅ Standard works | ❌ Enterprise required |

**Summary:** Cortex Analyst is cheaper below ~200 queries/day (zero infra, zero LLM setup). SnowMCP with Ollama wins at any meaningful scale. SnowMCP with Claude API reaches parity around 500–1,000 queries/day and stays cheaper above that — while Cortex Enterprise costs 3–6× more at every volume.

## 9. Security

| Aspect | SnowMCP | Snowflake Cortex Analyst |
|--------|---------|--------------------------|
| Credentials storage | `.env` file (local) / Kubernetes Secret / GitHub Secrets | Snowflake key pair or OAuth — managed by Snowflake |
| Data egress | Snowflake credentials leave your cluster; results are in-memory | Data stays within Snowflake — zero egress |
| Row / column level security | Enforced by Snowflake connector — inherited | Natively enforced by Snowflake RBAC |
| LLM data privacy | Depends on model: Ollama = fully local; Claude = Anthropic API | Snowflake-hosted models — data subject to Snowflake DPA |
| Audit logging | Not built-in — Kubernetes logs | Full Snowflake audit log via `QUERY_HISTORY` |
| Network isolation | Kubernetes NetworkPolicies | Snowflake private link / VPC peering |
| Secret rotation | Manual `.env` / Secret update | Snowflake key pair rotation |

**Cortex Analyst edge:** For enterprises requiring zero data-egress to third-party LLMs, Cortex Analyst is the safer choice since queries never leave Snowflake's perimeter.

**SnowMCP edge:** With Ollama, queries also never leave your infrastructure — matching Cortex on data privacy while giving full model control.

## 13. When to choose each

### Choose Cortex Analyst when:
- Your entire data estate lives in Snowflake — no cross-source needs
- You need enterprise audit trails with zero data leaving Snowflake
- Business users need a polished, click-to-deploy NL interface (Streamlit in Snowflake)
- You want zero infrastructure to manage
- Verified queries are critical — certain questions must always return the same SQL
- Your team has no engineering bandwidth to maintain a custom service

### Choose SnowMCP when:
- You need to query Snowflake **and** local files / other sources in the same session
- You want Claude or a local Ollama model instead of Snowflake's hosted model
- You need full SQL prompt transparency and control
- You want the semantic model in git with CI, code review, and rollback
- You're building agentic workflows that go beyond single-turn NL → SQL
- You need to run offline or in an air-gapped environment
- You're already paying for Claude Code — $0 additional cost
- You want to avoid Snowflake Enterprise tier requirements
- Your query volume is high enough that per-query Snowflake credit costs add up
